## Design constraint: BitNet as a bounded mode (Isaac, 2026-07-11)
- BitNet server = `llama-server.exe` (`B:\ai\llm\kitbash\bitnet\build\bin`) running
  `Little-Bitch-3B.i1-Q6_K.gguf` (2.2GB) on `127.0.0.1:8080`.
- **BitNet runs CPU-ONLY.** The llama.cpp build here is CPU-only (no `cuda`/`cublas` DLLs in the
  build tree — only `ggml.dll`/`llama.dll`/`llava_shared.dll`), so the `-ngl 20` flag is
  **silently ignored** — no layers are offloaded. Verified live: GPU memory used is ~788 MiB with
  the server running vs ~792 MiB with it killed → BitNet adds ≈0 to VRAM. This is the *intended*
  mode: BitNet's whole point is efficient low-bit CPU inference.
- **Stable config: `-c 4096`** (the `-ngl` value is a no-op on this build; keep it or drop it).
  Client defaults: `bitnet_max_tokens=256`, engine timeout 60s (256 tok ≈ 32s).
- **Memory budget is SYSTEM RAM, not VRAM.** BitNet footprint = weights ~2.2GB + KV-cache
  (~1280 MiB at `-c 4096`), all in CPU RAM. Since it touches ~0 VRAM, `-c` can go high freely
  (only costs RAM). The GPU (~2.1GB+ free) is entirely available for the eventual quantized
  synthesis LLM — **no VRAM contention** between BitNet and a GPU synthesis model.
- **Decision:** this config is stable → make it a **defined mode of operation** (fallback +
  efficiency), not the only mode.
- **HARD CONSTRAINT (revised):** the thing to watch is **system RAM** if BitNet + BitMamba + a
  CPU-side synthesis model ever run together. If the synthesis LLM runs on GPU (CUDA), it has the
  card to itself. (Putting BitNet on GPU would require rebuilding llama.cpp with CUDA — deferred,
  and arguably pointless given BitNet is meant to be the CPU-efficient tier.)
